# RETHREAD — 쓰레드 아카이브 (프로젝트 스펙 v2)

> 쓰레드(Threads)에서 리포스트·저장한 글을 다시 찾아보기 쉽게 아카이빙하는 로컬 우선(local-first) 웹앱.
> 핵심 원칙: **저장은 3초, 분류는 나중에, 재발견이 목적.**
>
> v2 변경점: 기존 프로토타입(threads-clipper)에서 검증된 **읽기 상태(status) 워크플로우**를 통합.
> 항목은 이제 두 개의 독립 축으로 관리된다 — **상태 축**(어디까지 읽었나) × **태그 축**(무슨 내용인가).

---

## 1. 제품 개요

### 해결하는 문제
- 쓰레드의 저장/리포스트 목록은 시간순 나열뿐이라 쌓일수록 다시 찾기 불가능
- 저장한 글이 삭제되거나 계정이 사라지면 내용 자체가 유실됨
- "저장만 하고 다시 안 보는" 문제 — 소비 워크플로우와 재발견 장치가 없음

### 해결 방식
1. 링크 붙여넣기 한 번으로 저장 (작성자·본문·썸네일 자동 수집 + 텍스트 스냅샷 영구 보관)
2. AI가 태그 1~3개 자동 제안, 사용자는 확인/수정만 (태그는 언제든 수정 가능)
3. 읽기 상태(나중에 볼 것 → 읽는 중 → 다 봄)로 소비 진행을 관리
4. 태그 필터 + 상태 필터 + 전문 검색 + 랜덤 재발견으로 다시보기

### 두 축의 역할 구분 (설계의 핵심)
| 축 | 질문 | 값 | 푸는 문제 |
|---|---|---|---|
| **상태 (status)** | 어디까지 읽었나 | later / reading / done | 저장만 하고 안 읽는 문제 |
| **태그 (tags)** | 무슨 내용인가 | 자유 태그 (AI 제안) | 쌓인 뒤 다시 못 찾는 문제 |

즐겨찾기(isFavorite)는 두 축과 별개의 "중요" 마커로 유지.

### 대상 사용자
- 1차: 본인 (single user, 로그인 없음)
- 향후: 각 사용자가 자기 브라우저에 데이터를 보관하는 형태로 공개 서비스화

## 2. 아키텍처

### 저장 전략: 로컬 우선 (Local-first)
- **모든 사용자 데이터는 브라우저 IndexedDB에 저장** (Dexie.js 사용)
- 서버 DB 없음. 회원가입 없음. 관리자 모드 없음. 서버는 사용자 데이터를 일절 저장하지 않음
- (v1 프로토타입의 공유 Gist + 관리자 구조는 폐기 — 다중 사용자 서비스화와 충돌)
- **백업**: JSON 내보내기/가져오기 필수 구현 (유일한 데이터 보험)
- 향후 확장: Supabase를 "선택적 동기화 계층"으로 추가 가능하도록 저장 로직을 repository 패턴으로 추상화

### 서버리스 함수 (무상태, 데이터 저장 안 함)
| 엔드포인트 | 역할 | 이유 |
|---|---|---|
| `POST /api/og` | 쓰레드 링크의 OG 메타태그 스크래핑 (작성자, 본문, 썸네일) | 브라우저 CORS 제한 우회 |
| `POST /api/suggest-tags` | Claude API로 본문 분석 → 태그 1~3개 제안 | API 키 서버 보관 (클라이언트 노출 금지) |

### 기술 스택
- **Next.js 14+ (App Router)** — 프론트 + API Route Handler 통합
- **Dexie.js** — IndexedDB 래퍼 (인덱스 기반 쿼리, 전문 검색 지원)
- **Tailwind CSS** — 스타일링
- **Anthropic SDK** — 태그 제안 (`claude-haiku-4-5` 권장: 저렴하고 충분)
- **배포: Vercel** — 환경변수 `ANTHROPIC_API_KEY`

## 3. 데이터 모델 (IndexedDB, Dexie 스키마)

```ts
type ReadStatus = 'later' | 'reading' | 'done';

interface ArchiveItem {
  id: string;              // nanoid
  url: string;             // 원본 쓰레드 링크
  author: string;          // @handle
  authorName?: string;     // 표시 이름
  previewText: string;     // 본문 스냅샷 (OG description 전문 — 원본 삭제 대비 영구 보관)
  thumbnail?: string;      // OG 이미지 URL (선택: base64로 로컬 보관 옵션)
  tags: string[];          // 사용자 확정 태그 (AI 제안 → 사용자 수정 가능)
  status: ReadStatus;      // 읽기 상태. 기본값 'later'
  memo?: string;           // 사용자 메모 ("왜 저장했는지")
  isFavorite: boolean;     // 중요 마커 (상태·태그와 독립)
  isClassified: boolean;   // false = 인박스에 노출 (태그 확정 전)
  savedAt: number;         // epoch ms
  statusChangedAt: number; // 상태 변경 시각 (재발견 로직에 사용)
  lastViewedAt?: number;   // 재발견 로직에 사용
}

interface TagMeta {
  name: string;            // 태그명 (primary key)
  color?: string;          // 사용자 지정 색
  count: number;           // 캐시된 사용 횟수
}
```

**Dexie 인덱스**: `id`, `savedAt`, `status`, `isClassified`, `isFavorite`, `*tags` (multiEntry)

**상태 전이 규칙**
- 생성 시 `status: 'later'`
- 상세 시트에서 원본 링크를 열면 `'later' → 'reading'` 자동 전환 (이미 done이면 유지)
- `'done'` 전환은 사용자 명시 액션만 (자동 전환 없음)
- 모든 상태 변경 시 `statusChangedAt` 갱신

## 4. 화면 구조 (3개 + 설정)

### 4.1 인박스 (홈)
- 상단 고정: URL 붙여넣기 입력창 (붙여넣기 즉시 자동 저장 시작)
- 저장 플로우:
  1. URL 입력 → `/api/og` 호출 → 카드 미리보기 즉시 표시 (낙관적 UI)
  2. 동시에 `/api/suggest-tags` 호출 → 제안 태그가 칩으로 표시 (점선 테두리 = 미확정)
  3. 사용자는 **탭 한 번으로 태그 확정** / X로 제거 / 직접 입력 추가
  4. "보관" 시 `isClassified: true` → 라이브러리로 이동 (status는 'later' 유지). 건너뛰면 인박스에 잔류
- 인박스 목록: 미분류(`isClassified: false`) 항목만 시간 역순
- OG 스크래핑 실패 시: URL만 저장 + 수동 입력 폼 fallback (저장 자체는 절대 실패하지 않게)

### 4.2 라이브러리
- 상단: 검색창 (작성자·본문·메모·태그 대상 클라이언트 전문 검색)
- **상태 세그먼트 탭** (전체 / 나중에 볼 것 / 읽는 중 / 다 봄) — v1 프로토타입 UX 계승, 항상 노출되는 1차 필터
- 태그 칩 가로 스크롤 필터 (다중 선택 = AND 조건) + 즐겨찾기 토글 — 2차 필터, 상태 필터와 AND 결합
- 카드 그리드: 썸네일 + 작성자 + 본문 2줄 + 태그 칩 + 상태 뱃지(색상 점)
- 카드의 상태 뱃지 탭 → 인라인으로 상태 순환 변경 (later → reading → done → later)
- 카드 탭 → 상세 시트:
  - 본문 전문, 저장일, 원본 링크 열기 (열면 later→reading 자동 전환)
  - 상태 선택 세그먼트, 메모 편집 (blur 시 자동 저장)
  - **태그 편집(추가/삭제/AI 재제안 버튼)**, 즐겨찾기 토글, 삭제
- 정렬: 최신순(기본) / 오래된순 / 즐겨찾기 우선

### 4.3 다시보기 (재발견)
- **"읽다 만 글"** 섹션: status='reading'인데 statusChangedAt이 7일 이상 지난 항목 (소비 재개 유도 — 최우선 노출)
- **"오늘의 발견"**: lastViewedAt이 오래됐거나 없는 done/later 항목 가중 랜덤 3개
- **"그때 그 글"**: 1/3/6/12개월 전 ±3일 범위에 저장된 항목
- 카드를 보면 `lastViewedAt` 갱신

### 4.4 설정
- **JSON 내보내기** (전체 데이터 → `rethread-backup-YYYYMMDD.json` 다운로드)
- **JSON 가져오기** (병합: id 기준 중복 제거 / 덮어쓰기 선택)
- **v1 마이그레이션**: threads-clipper의 Gist JSON을 가져오기에서 인식해 필드 매핑 (상태값 매핑 포함, 중요 표시 → isFavorite)
- 태그 관리 (이름 변경 → 전체 항목 일괄 반영, 병합, 삭제)
- 데이터 통계 (총 항목 수, 상태별 분포, 태그 수, 저장 용량)
- 백업 리마인더: 마지막 내보내기 30일 경과 시 배지

## 5. 태그 자동 제안 로직

```
POST /api/suggest-tags
body: { text: string, existingTags: string[] }
```

- Claude에게 본문 + **사용자의 기존 태그 목록**을 함께 전달
- 지시: "기존 태그 중 맞는 것을 우선 재사용하고, 없을 때만 새 태그 제안. 한국어, 1~3개, JSON 배열만 반환"
- 기존 태그 재사용을 우선하는 이유: 태그 파편화 방지 (예: '투자'와 '재테크' 난립 방지)
- 응답 파싱 실패 시 빈 배열 반환 → UI는 수동 입력으로 자연스럽게 fallback
- **제안은 제안일 뿐**: 사용자가 확정하기 전까지 저장되지 않으며, 확정 후에도 상세 시트에서 언제든 수정 가능

## 6. 디자인 방향

- 모바일 우선 (쓰레드 앱에서 링크 복사 → 바로 붙여넣는 동선)
- 다크 모드 기본 (쓰레드와 톤 일치), 미니멀
- 카드 중심 레이아웃. 태그 칩은 색상, 상태는 점 뱃지 색으로 구분 (later=회색, reading=파랑, done=초록, 중요=노랑 별)
- PWA manifest 포함 (홈 화면 추가 시 앱처럼 사용)

## 7. 명시적 비범위 (Non-goals)

- 로그인/회원가입 없음
- 서버측 사용자 데이터 저장 없음, 관리자 모드 없음 (v1의 공유 Gist 구조 폐기)
- Threads API 연동 없음 (공식 API가 저장 목록을 제공하지 않음 — 수동 링크 붙여넣기가 유일한 현실적 입력)
- 소셜/공유 기능 없음

## 8. 향후 확장 (v2 이후)
